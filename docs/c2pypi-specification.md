# c2pypi Specification

## Motivation

c2py23 solves two things: generating CPython C wrappers from a `.c2py` YAML
file, and binary portability across Python 2.7 through 3.15 via the nimpy
trick (all CPython API resolved at module load time via `dlopen(NULL)` +
`dlsym()`).  Verified: a `.so` built on CentOS 7 (manylinux2014, glibc 2.17)
with Python 3.12 loads on Python 2.7 through 3.14t across Debian, Ubuntu
20.04, and Ubuntu 24.04.

What c2py23 does NOT solve is the wheel / packaging / PyPI distribution
story.  That problem has its own long history (wheel tagging rules,
compiler management, manylinux policies, cross-compilation, build-backend
integration, `twine check` validation, CI for multi-architecture builds)
and its own stakeholders.  It is a separate project.

c2pypi is that project.  It is a **packer**.  It takes pre-built `.so` files
(one per architecture), arranges them into a versioned-subfolder layout,
generates an import trampoline, and assembles a `py3-none-any` wheel that
pip can install on any platform.

c2pypi does NOT manage compilers.  It does not invoke `gcc`, `clang`, or
`icx`.  It does not know about `-mavx512f`, `-fPIC`, or `-O3`.  All of that
belongs to the user's build system -- setuptools, meson, CMake, a Makefile,
or `c2py23 build` itself.

**What already exists in c2py23:** The `c2py_loader.py` module (`c2py23/c2py_loader.py`)
implements the explicit-filename loading convention (`_module.c2py23-{os}_{arch}.so`)
described in this spec.  It loads `.so` files by full path via `ExtensionFileLoader`
(3.x) or `imp.load_dynamic` (2.7), with `C2PY_TRACE` instrumentation for debugging.
The `examples/wheel_demo/`, `examples/meson_demo/`, and `examples/cmake_demo/`
directories demonstrate the full build + wheel packaging flow using this convention.
c2pypi (future project) would use the same loader internally and add packer /
multi-multi-arch CI orchestration around it.

## Relationship to c2py23

c2pypi and c2py23 are orthogonal.  c2py23 is a code generator (`c2py23
generate mymodule.c2py` produces a wrapper C file).  c2pypi is a packer
(`c2pypi pack _native.so` produces a wheel).  Neither imports the other.
The interface is a file on disk -- the compiled `.so`.

```
                      c2py23 domain
                    +---------------+
  mymodule.c2py --->| c2py23        |
  mymodule.c   --->|   generate     |---> wrapper.c
                    +---------------+
                            |
                    +---------------+
                    | build system  |
                    | (meson, make, |
                    |  setuptools,  |
                    |  c2py23 build)|    user's choice
                    +---------------+
                            |
                        _native.so
                            |
                    +---------------+
                    | c2pypi        |
                    |   pack        |    arch-subfolder
                    |   verify      |    wheel assembly
                    +---------------+
                            |
                    py3-none-any.whl
```

## Core Design Principle: punting on the compiler driver

c2pypi accepts already-compiled `.so` files.  It never runs a compiler.
This decision is informed by the long history of projects that tried to
own the compiler story and regretted it (f2py's Fortran compilers,
numpy.distutils).

The user's build system -- whatever they choose -- handles:
- Which compiler (gcc / clang / icx / mingw / ...)
- Which flags (`-O3`, `-mavx512f`, `-fPIC`, `-fopenmp`, ...)
- Cross-compilation (`aarch64-linux-gnu-gcc`, `x86_64-w64-mingw32-gcc`)
- SIMD variant compilation (multiple `.o` files with different ISA flags)
- Linking (`-ldl`, `-lm`, static vs shared)

c2pypi's only concern: "here is a directory of `.so` files, one per
architecture.  Make a wheel."

## The Arch-Subfolder Layout

The wheel is tagged `py3-none-any` and contains:

```
mymodule/
    __init__.py                      # generated by c2pypi
    _arch/
        linux_x86_64/_native.so      # compiled inside manylinux2014
        linux_aarch64/_native.so     # cross-compiled or native-built
        linux_ppc64le/_native.so     # ditto
        win_amd64/_native.pyd        # cross-compiled via mingw (future)
        darwin_x86_64/_native.so     # built on macOS
        darwin_arm64/_native.so      # built on macOS
```

Subfolder names are `{platform}_{machine}` with normalization:

| Raw `sys.platform` | Normalized | Raw `platform.machine()` | Normalized |
|---|---|---|---|
| `linux2`, `linux` | `linux` | `AMD64` | `x86_64` |
| `win32` | `win` | `aarch64` | `aarch64` |
| `darwin` | `darwin` | `arm64` | `arm64` |
| | | `ppc64le` | `ppc64le` |

The `.so` inside each subfolder is always named `_native` (or `_native.pyd`
on Windows).  The c2py23 `.c2py` file uses `module: _native`.  This means
the compiled `.so` for every architecture is byte-identical -- same init
function, same module name -- only the enclosing folder path differs.

## The Import Trampoline

`__init__.py` is generated by c2pypi.  It detects the platform at import
time, selects the right subfolder, and loads the `.so`:

```python
import os as _os, sys as _sys, platform as _platform, importlib as _importlib

def _c2py_load():
    _plat = _sys.platform
    if _plat == 'linux2':
        _plat = 'linux'
    _arch = _platform.machine()
    if _arch == 'AMD64':
        _arch = 'x86_64'
    _key = '%s_%s' % (_plat, _arch)

    _here = _os.path.dirname(_os.path.abspath(__file__))
    _arch_dir = _os.path.join(_here, '_arch', _key)

    if not _os.path.isdir(_arch_dir):
        _supported = _os.listdir(_os.path.join(_here, '_arch'))
        raise ImportError(
            "c2pypi: no native module for '%s'.  "
            "Supported platforms: %s" % (_key, ', '.join(sorted(_supported))))

    _sys.path.insert(0, _arch_dir)
    try:
        _mod = _importlib.import_module('_native')
    finally:
        _sys.path.pop(0)

    for _k, _v in _mod.__dict__.items():
        if not _k.startswith('_'):
            globals()[_k] = _v

_c2py_load()
del _c2py_load
```

This trampoline has been verified on:
- CPython 2.7 (ubuntu20.04) -- all primitives available
- CPython 3.6 (debian10)
- CPython 3.9, 3.12 (ubuntu24.04)
- CPython 3.14 (ubuntu26.04)
- CPython 3.14t free-threaded (ubuntu26.04)

Why it works on free-threaded Python: `sys.path` manipulation happens
during import, before the module is exposed to user threads.  No lock
needed.  The trampoline makes no assumption about GIL state.

### Python version gating

Before the dynamic import, the trampoline checks for unsupported Python
versions (3.16+ before the ABI has been audited):

```python
if _sys.version_info >= (3, 16):
    raise ImportError(
        "Python %d.%d is not yet supported by c2pypi.  "
        "Supported: 2.7, 3.6-3.15." % _sys.version_info[:2])
```

Version bounds are configurable in the trampoline template.  c2pypi
supplies a default that matches the tested range.

### Disclaimer for unsupported implementations

c2pypi-packaged modules rely on `dlopen(NULL, RTLD_GLOBAL)` at module init
time to resolve CPython API symbols.  This requires the Python interpreter
to be built with `--enable-shared` (or equivalent, so that CPython symbols
are exported from the process).  This is true for most Linux distribution
Pythons and the python.org macOS installers.  It is NOT guaranteed on:
- PyPy (different C API surface)
- Some conda builds
- Embedded/frozen Python interpreters
- Python statically linked (no dynamic symbol table)

The trampoline catches `ImportError` from the `.so` load and reports a
clear diagnostic rather than an opaque linker error.

## Commands

```
c2pypi pack [options] <so-files...>
    Assemble pre-built .so files into a py3-none-any wheel.

    Options:
      --module <name>           Module name (default: from pyproject.toml)
      --version <ver>           Version string (default: from pyproject.toml)
      --platform <os>_<arch>    Override platform detection (for CI)
      --output-dir <dir>        Output directory (default: dist/)
      --flat                    Use flat naming (_c2py_{os}_{arch}.so)
                                instead of subfolder layout

    If no .so files are given, c2pypi scans the current directory for
    _arch/{os}_{arch}/_native.{so,pyd} or _c2py_*.{so,pyd}.

c2pypi verify <wheel>
    Test a built wheel on the current Python.

    Loads the module, checks that the trampoline resolves the correct
    architecture, and runs a smoke test (import + basic sanity).  Does
    NOT run the full test suite -- the user's CI handles that.

c2pypi init <dir>
    Scaffold a new c2pypi project directory with pyproject.toml,
    example layout, and .gitignore.
```

### Example: single-arch build

```bash
# Build .so with the user's preferred build system
c2py23 generate mymodule.c2py -o wrapper.c
gcc -shared -fPIC -I $C2PY/runtime wrapper.c mymodule.c runtime.c -ldl -lm \
    -o _native.so

# Pack
c2pypi pack --module mymodule --version 0.1.0 --platform linux_x86_64 _native.so
```

### Example: multi-arch CI build

```bash
# In CI, build per architecture inside containers, collect results:
for arch in linux_x86_64 linux_aarch64; do
    build_container $arch  # produces _native.so in arch subfolder
    mkdir -p _arch/$arch
    cp build-$arch/_native.so _arch/$arch/
done

# Pack everything into one wheel
c2pypi pack --module mymodule --version 0.1.0
```

c2pypi auto-detects `_arch/*/_native.so` and includes them.

## The pyproject.toml config

c2pypi reads packaging metadata from `pyproject.toml`.  It does NOT read
compiler flags, SIMD flags, or build instructions from this file -- those
belong in the user's build system.

```toml
[build-system]
# For sdist / source installs: use whatever build system the user wants.
# c2pypi is NOT the build backend for sdist.
requires = ["setuptools>=44"]   # or ["meson-python"], or ["c2py23", "setuptools"]
build-backend = "setuptools.build_meta"

[project]
name = "mymodule"
version = "0.1.0"

[tool.c2pypi]
# Minimal: just module name.  Version from [project].
module = "mymodule"

# Subfolder layout (default, no config needed):
layout = "subfolder"

# Alternative: flat naming
# layout = "flat"
# prefix = "_mymod"          # results in _mymod_linux_x86_64.so etc.

# Python version bounds for the trampoline (optional, defaults shown):
trampoline.min-python = "2.7"
trampoline.max-python = "3.15"

# Platforms with known unsupported status (reported in trampoline error):
unsupported = ["musllinux_1_1_x86_64", "win_arm64"]
```

Note: `[tool.c2pypi]` is used by `c2pypi pack` for defaults when `--module`
and `--version` are not passed on the command line.  It is NOT consulted
during `pip install` of the resulting wheel.

## sdist and pip install from source

c2pypi binary wheels are the fast path.  For source installs, the user
provides their own build system in `pyproject.toml`:

```toml
[build-system]
requires = ["c2py23", "setuptools>=44"]
build-backend = "setuptools.build_meta"
```

At `pip install` from source:
1. pip fetches build dependencies (c2py23)
2. The build backend (setuptools) runs `c2py23 generate` to produce the
   wrapper, then compiles `_native.so` for the host architecture only
3. The installed module includes just one architecture in the subfolder

c2pypi is NOT involved in `pip install` from source.  This keeps the
separation clean: c2pypi is for building portable binary wheels; the
user's build system is for source installs.

For projects that want `pip install` from source to produce the full
multi-arch layout: they can include `c2pypi` in their sdist and run
`c2pypi pack` as a post-install hook.  But this is unusual -- most
users who `pip install` from source want a single-arch build that
matches their host.

### Python 2.7

The `py3-none-any` wheel tag excludes Python 2.7.  Python 2.7 users
install from source via `pip install --no-binary mymodule mymodule`.
The sdist contains everything needed to compile `_native.so` for 2.7,
and c2py23's generated wrapper is already 2.7-compatible.

## Flat Naming Alternative

As an alternative to the subfolder layout, c2pypi also supports a
flat naming convention:

```
mymodule/
    __init__.py
    _mymod_linux_x86_64.so
    _mymod_linux_aarch64.so
    _mymod_linux_ppc64le.so
```

In flat mode, the `.so` module name must match the filename.  This requires
compiling with a different `module:` per architecture (e.g., `module:
_mymod_linux_x86_64` vs `module: _mymod_linux_aarch64`).  c2pypi provides a
`--rename-init` option that patches the `.so` to match the filename.

The trampoline with flat naming:

```python
_name = '_mymod_%s_%s' % (_plat, _arch)
_mod = _importlib.import_module('mymodule.%s' % _name)
# re-export
```

Flat naming avoids `sys.path` manipulation, making the trampoline a
trivial dotted import.  The tradeoff is that the build system must
produce architecturally-named `.so` files.

Recommendation: start with subfolder layout.  Add flat naming when
the build system integration story is mature enough to support
per-arch module naming.

## Compiler Management (delegated)

c2pypi does not manage compilers.  How the user's build system obtains
and invokes compilers is the user's concern.  Typical patterns:

### Pattern A: Container-per-arch

```bash
# Build inside a manylinux2014 container for x86_64
apptainer exec manylinux2014.sif /bin/bash -c "
    gcc -shared -fPIC ... -o _native.so
"
mkdir -p _arch/linux_x86_64 && mv _native.so _arch/linux_x86_64/
```

### Pattern B: Cross-compilation from one container

```bash
apptainer exec manylinux2014.sif /bin/bash -c "
    # Native x86_64
    gcc -shared -fPIC ... -o _native_x86_64.so

    # Cross-compile aarch64
    aarch64-linux-gnu-gcc -shared -fPIC ... -o _native_aarch64.so

    # Cross-compile ppc64le
    powerpc64le-linux-gnu-gcc -shared -fPIC ... -o _native_ppc64le.so
"
```

### Pattern C: SIMD multi-flag compilation

The user's build system compiles each SIMD variant as a separate `.o`
file, then links them into one `.so` per architecture.  The dispatch
logic (which variant to call at runtime) is generated by c2py23 and
compiled into the wrapper.  c2pypi never sees the SIMD flags -- they
stay in the user's Makefile or `meson.build`.

```makefile
# User's Makefile (SIMD flags live here, not in pyproject.toml)
poly_f32_avx512.o: poly_kernel.c
    gcc -c -O3 -mavx512f -DKERNEL_FN=poly_f32_avx512 $< -o $@

poly_f32_avx2.o: poly_kernel.c
    gcc -c -O3 -mavx2 -DKERNEL_FN=poly_f32_avx2 $< -o $@

poly_f32_scalar.o: poly_kernel.c
    gcc -c -O3 -DKERNEL_FN=poly_f32_scalar $< -o $@

_native.so: wrapper.c mymodule.c poly_f32_avx512.o poly_f32_avx2.o poly_f32_scalar.o
    gcc -shared -fPIC $^ -ldl -lm -o $@
```

The `.c2py` file declares the variants, c2py23 generates the dispatch
wrapper, the Makefile compiles and links, c2pypi packs the result.
Clean separation end to end.

## CI Integration

c2pypi is designed to be called from CI scripts.  It does not provide
CI templates -- those live in the user's repository.

Example GitHub Actions workflow:

```yaml
jobs:
  build-linux-x86_64:
    runs-on: ubuntu-latest
    container: quay.io/pypa/manylinux2014_x86_64
    steps:
      - uses: actions/checkout@v4
      - run: pip install c2py23
      - run: c2py23 generate mymodule.c2py -o wrapper.c
      - run: make _native.so
      - run: mkdir -p _arch/linux_x86_64 && cp _native.so _arch/linux_x86_64/
      - uses: actions/upload-artifact@v4
        with: {name: so-linux-x86_64, path: _arch/}

  build-linux-aarch64:
    runs-on: ubuntu-latest
    container: quay.io/pypa/manylinux2014_x86_64
    steps:
      - run: yum install -y gcc-aarch64-linux-gnu
      # ... same pattern, cross-compile for aarch64
      - uses: actions/upload-artifact@v4
        with: {name: so-linux-aarch64, path: _arch/}

  pack:
    needs: [build-linux-x86_64, build-linux-aarch64]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
      - run: pip install c2pypi
      - run: c2pypi pack --module mymodule --version ${{ github.ref_name }}
      - uses: actions/upload-artifact@v4
        with: {name: wheel, path: dist/*.whl}
      - run: twine check dist/*.whl
      - uses: pypa/gh-action-pypi-publish@release/v1
        with: {packages-dir: dist/}

  test:
    needs: pack
    strategy:
      matrix:
        container: [ubuntu24.04.sif, debian10.sif, ubuntu20.04.sif]
        python: ["3.6", "3.9", "3.12", "3.14t"]
    steps:
      - run: pip install dist/*.whl
      - run: python3 -c "import mymodule; print(mymodule.__doc__)"
      - run: python3 -m pytest tests/
```

## Free-Threaded Python

The trampoline does not assume GIL availability.  It performs `sys.path`
manipulation and `importlib.import_module()` during module initialization,
before the module object is exposed to user code.  Under free-threading,
this is single-threaded by construction.

The `.so` loaded by the trampoline may or may not declare
`Py_MOD_GIL_NOT_USED`.  If it does, the GIL stays disabled.  If it does
not, CPython 3.14+ re-enables the GIL for that module's calls.  The
trampoline code does not need to know which case applies -- it is a
pass-through loader.

The per-function `gil_release` and `free_threading` settings in the
user's `.c2py` file control the `.so`'s behavior; c2pypi is not involved.

## Testing Model

c2pypi provides `c2pypi verify` for smoke testing a wheel.  For
full testing, the user's CI runs the wheel's test suite inside
containers for each Python version.

Cross-architecture testing:

| Built for | Test method |
|-----------|-------------|
| `linux_x86_64` | Native import in manylinux2014 or any Linux container |
| `linux_aarch64` | `qemu-aarch64-static` + Linux container |
| `linux_ppc64le` | `qemu-ppc64le-static` + Linux container |
| `win_amd64` | Native Windows CI runner or Wine |
| `darwin_x86_64` | Native macOS CI runner |
| `darwin_arm64` | Native macOS CI runner |

### What to verify per platform

1. The wheel installs (`pip install` succeeds)
2. `import mymodule` loads the correct arch subfolder (check `_native.__file__`)
3. The test suite passes (user-supplied)
4. No refcount leaks (c2py23's error path stress test)

## Open Questions

### auditwheel / delocate

auditwheel inspects ELF dependencies in platform-specific wheels and
can add missing libraries or adjust RPATH.  For `py3-none-any` wheels:
these tools are not normally invoked because the wheel declares no
platform tag.  However, if the wheel is later re-tagged to a
platform-specific tag (e.g., by auditwheel itself), the `.so` files
inside may be scanned.

For ctypes-loaded `.so` files in other projects: auditwheel's
`repair` command scans ALL ELF files in the wheel, including those
not declared as extension modules.  c2pypi-packaged wheels are
`py3-none-any` so this path is not triggered, but it should be
documented as a known interaction if users re-tag their wheels.

### musllinux (Alpine)

The `.so` files built on manylinux2014 (glibc 2.17) will not load on
musl-based systems.  musllinux wheels require building against musl
libc.  This is a separate build target, not a c2pypi issue -- the user
needs a musl-based container that produces `_native.so` for `musl_x86_64`.
c2pypi supports it by adding the subfolder: `_arch/musl_x86_64/_native.so`.

No work is needed in c2pypi to support musllinux -- just a different
build container.

### Windows

Two separate concerns:

1. **c2py23 runtime**: The nimpy trick on Windows requires replacing
   `dlopen(NULL)` with `GetModuleHandle(NULL)` and `dlsym()` with
   `GetProcAddress()`.  This is a c2py23 runtime change, not c2pypi.

2. **PEP 517 tooling**: Windows CI runners, MSVC vs clang vs mingw,
   `.pyd` vs `.dll` naming.  c2pypi supports `.pyd` as the extension
   in subfolders (alongside `.so`).

Windows port is planned before c2pypi's first release because working
through the Windows build story forces the separation between binding
generator and packer to be real, not aspirational.

### Python version ceiling

The trampoline version gate (3.16+) is a runtime check, not a pip
constraint.  pip's `Requires-Python` metadata in the wheel can
additionally gate at install time:

```toml
[project]
requires-python = ">=2.7,<3.16"
```

This prevents pip from installing the wheel on Python 3.16+ before
the ABI has been audited for that version.

### Build isolation with c2py23

For sdist / source installs where the wrapper is regenerated:
- c2py23 must be in `build-system.requires`
- pip fetches and installs c2py23 into an isolated build environment

For binary wheels (`py3-none-any.whl`):
- The wrapper is pre-generated, embedded in the `.so`
- No c2py23 is needed at install time
- `build-system.requires` can be empty or absent

The CI build step (on the developer's machine or in CI) always needs
c2py23 to generate the wrapper before compilation.  The end user
installing the wheel needs nothing beyond pip.

## Reference Implementation Plan

### Phase 1: Linux amd64 single-arch

- `c2pypi pack --platform linux_x86_64 _native.so`
- Generates `__init__.py` trampoline (subfolder layout)
- Assembles `py3-none-any.whl` using the `wheel` library
- `c2pypi verify` smoke-tests the wheel
- Validate with `twine check`

### Phase 2: Multi-arch collect + CI

- Auto-detect `_arch/*/_native.so` from directory structure
- Generate trampoline with multiple supported platforms listed in error
- GitHub Actions CI template
- Cross-compilation build scripts (as documentation, not packaged code)

### Phase 3: Flat naming + rename-init

- `--flat` mode with `_c2py_{os}_{arch}.so` naming
- `--rename-init` to patch the `.so`'s exported init symbol
- Build system integration documentation (meson, cmake, setuptools)

### Phase 4: Windows + macOS

- Windows CI runner support
- `.pyd` extension handling
- macOS universal2 binary support (future)

### Phase 5: c2py23 strips build/compile

- `c2py23 build` deprecated in favor of `c2py23 generate` + user's build system
- `c2py23 compile` removed
- c2py23's own test suite (`test_all.py`, `test_manylinux.py`) refactored
  to use a build script + c2pypi
